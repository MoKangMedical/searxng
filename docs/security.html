Defense in Depth

Five-layer security model from network perimeter to data storage

Layer 1: Network Perimeter Nginx WAF, DDoS Protection, IP Allowlist
Layer 2: Transport Security TLS 1.3, HSTS, Certificate Pinning
Layer 3: Application Security CSRF, XSS Prevention, Content Security Policy
Layer 4: Data Protection Redis AUTH, Encrypted Volumes, No PII Storage
Layer 5: Access Control API Key Auth, Rate Limiting, Audit Logging

Security Features

Comprehensive security controls protecting every component

Privacy by Design

Zero-knowledge architecture: no user tracking, no search history storage, no IP logging. SearXNG's core philosophy is privacy-first meta-search. All queries are anonymized before reaching upstream engines.

Active

TLS 1.3 Encryption

All external connections use TLS 1.3 with strong cipher suites. Internal Docker network communication is isolated. HSTS headers enforce HTTPS-only access. Certificate auto-renewal via Let's Encrypt.

Active

Container Isolation

Docker containers run with minimal capabilities (CHOWN, SETGID, SETUID only). All other Linux capabilities are dropped. Read-only root filesystem where possible. Non-root user execution.

Active

Rate Limiting

Multi-tier rate limiting: Nginx level (100 req/min per IP), application level (20 req/min per API key), and engine level (5 concurrent per engine). Redis-backed counters with automatic expiry.

Active

Security Headers

Complete security header suite: Content-Security-Policy, X-Frame-Options, X-Content-Type-Options, Referrer-Policy, Permissions-Policy. Prevents clickjacking, MIME sniffing, and information leakage.

Active

Audit Logging

Structured JSON logging for all API requests, authentication events, and configuration changes. Log rotation with 30-day retention. Optional integration with ELK stack or Grafana Loki.

Configurable

Vulnerability Scanning

Automated dependency scanning via Dependabot. Container image scanning with Trivy. Weekly security audit reports. CVE monitoring for all upstream dependencies.

Active

Backup & Recovery

Automated daily backups of Redis data and configuration. Point-in-time recovery with 30-day retention. Disaster recovery runbook with RTO < 1 hour, RPO < 24 hours.

Configurable

Security Configuration

Production-ready security settings and best practices

Nginx Security Headers ▼ 
# Security Headers
add_header X-Frame-Options "SAMEORIGIN" always;
add_header X-Content-Type-Options "nosniff" always;
add_header X-XSS-Protection "1; mode=block" always;
add_header Referrer-Policy "strict-origin-when-cross-origin" always;
add_header Content-Security-Policy "default-src 'self'; script-src 'self' 'unsafe-inline'; style-src 'self' 'unsafe-inline'; img-src 'self' data: https:;" always;
add_header Permissions-Policy "geolocation=(), microphone=(), camera=()" always;
add_header Strict-Transport-Security "max-age=31536000; includeSubDomains" always;

# Rate Limiting
limit_req_zone $binary_remote_addr zone=api:10m rate=20r/m;
limit_req_zone $binary_remote_addr zone=search:10m rate=100r/m;
limit_conn_zone $binary_remote_addr zone=conn:10m;
Redis Security Configuration ▼ 
# redis.conf security settings
requirepass ${REDIS_PASSWORD}
rename-command FLUSHDB ""
rename-command FLUSHALL ""
rename-command DEBUG ""
rename-command CONFIG "CONFIG_b83a2c1e"
bind 127.0.0.1
protected-mode yes
maxmemory 256mb
maxmemory-policy allkeys-lru
  • Password authentication enabled
  • Dangerous commands disabled (FLUSHDB, FLUSHALL, DEBUG)
  • CONFIG command renamed for obscurity
  • Bound to localhost only (accessible via Docker network)
  • Memory limit with LRU eviction policy
Docker Security Hardening ▼ 
# docker-compose.yml security settings
services:
  searxng:
    cap_drop:
      - ALL
    cap_add:
      - CHOWN
      - SETGID
      - SETUID
    read_only: true
    tmpfs:
      - /tmp
    security_opt:
      - no-new-privileges:true
    deploy:
      resources:
        limits:
          memory: 512M
          cpus: '1.0'
  • All Linux capabilities dropped except minimum required
  • Read-only root filesystem with tmpfs for temp files
  • No privilege escalation allowed
  • Resource limits prevent DoS
Incident Response Procedure ▼
  1. Detection - Automated monitoring alerts (Prometheus/Grafana) or manual discovery
  2. Triage - Assess severity (P0-P3), identify affected components
  3. Containment - Isolate affected services, rotate credentials if needed
  4. Eradication - Remove root cause, patch vulnerabilities
  5. Recovery - Restore from backup, verify service integrity
  6. Post-mortem - Document lessons learned, update runbooks

Production Security Checklist

Pre-deployment security verification items

Infrastructure

  • Firewall rules configured
  • SSH key-only authentication
  • Automatic security updates enabled
  • Container images from trusted registries
  • Network segmentation implemented

Application

  • Secret key changed from default
  • Debug mode disabled
  • Error pages do not leak stack traces
  • API authentication enabled
  • Input validation on all endpoints

Monitoring

  • Log aggregation configured
  • Alert rules defined
  • Health check endpoints active
  • Uptime monitoring enabled
  • Penetration test scheduled
