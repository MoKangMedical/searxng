Component Architecture

Detailed breakdown of each system component and its responsibilities

SSearXNG Core

Meta-search engine that aggregates results from 30+ search engines. Handles query parsing, engine selection, result merging, and deduplication.

Python 3.11 Flask httpx lxml

RRedis Cache

In-memory caching layer with 24h TTL for search results. Also handles session management, rate limiting counters, and autocomplete suggestions.

Redis 7 Alpine AOF Persistence

NNginx Proxy

Reverse proxy handling SSL termination, request routing, static asset serving, and security headers. Provides rate limiting and DDoS protection.

Nginx 1.25 SSL/TLS gzip

AAI Summarizer

Optional LLM-powered module that summarizes search results, extracts key insights, and provides intelligent answers for medical queries.

LangChain OpenAI DeepSeek

QQuery Processor

Parses user queries, detects language, applies bang shortcuts (!bd, !gs), handles autocomplete, and routes to appropriate engine groups.

NLP Bang Syntax Language Detection

MMedical Search

Domain-specific search module for healthcare queries. Integrates with PubMed, ClinicalTrials.gov, and medical knowledge bases.

PubMed API DrugBank MeSH

Request Data Flow

How a search query flows through the system from client to response

1

Client Request

User submits search query via web UI or API. Request hits Nginx reverse proxy with optional SSL termination.

2

Rate Limit Check

Nginx checks rate limit counters in Redis. Excessive requests receive 429 Too Many Requests response.

3

Cache Lookup

SearXNG checks Redis cache for identical query. Cache hit returns stored results in <200ms, skipping engine requests entirely.

4

Query Parsing

Query processor parses the input, detects language (zh/en), extracts bang shortcuts (!bd for Baidu, !gs for Scholar), and selects target engines.

5

Concurrent Engine Requests

SearXNG sends parallel requests to selected engines (up to 5 concurrent per engine). Each request has a 5s timeout with automatic fallback.

6

Result Aggregation

Results from all engines are merged, deduplicated by URL, scored by relevance, and ranked. Engine metadata (source, reliability) is attached.

7

Cache Store & Response

Aggregated results are stored in Redis with 24h TTL. Final response is returned as HTML, JSON, CSV, or RSS format.

Network Isolation ▼

All services communicate through a dedicated Docker bridge network (searxng). No service exposes internal ports to the host except through Nginx.

Scalability Design

Horizontal and vertical scaling strategies for production workloads

Horizontal Scaling

Multiple SearXNG instances behind Nginx load balancer. Redis shared cache ensures consistent results across instances. Docker Swarm or Kubernetes for orchestration.

Cache Optimization

Three-tier caching: in-memory (Flask), Redis (shared), CDN (static assets). Cache warming for popular queries. TTL-based invalidation with manual purge API.

Engine Pool

Engine health monitoring with automatic failover. Unresponsive engines are temporarily suspended. Circuit breaker pattern prevents cascade failures.

Geographic Distribution

Deploy instances in multiple regions (China mainland, Hong Kong, US) with DNS-based routing. Each region has optimized engine selection for local results.
